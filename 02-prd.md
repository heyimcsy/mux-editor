# PRD — Mux Editor (Cmux / Wmux 테마 편집기)

> 원본 기획: `01-PRD.txt`

## 1. 요약 (Summary)

Mux Editor는 cmux(macOS)와 wmux(Windows) 터미널의 색상 테마를 브라우저에서 눈으로 보면서 만드는 웹 편집기입니다. 사용자는 가상 터미널 화면에 색을 바로 입혀보고, 마음에 드는 결과가 나오면 그대로 붙여넣을 수 있는 설정 파일과 명령어를 받아갑니다. 설정 파일 문법을 몰라도 자기 터미널을 자기 스타일로 꾸밀 수 있게 만드는 것이 목표입니다.

## 3. 배경 (Background)

### 지금 무슨 일이 벌어지고 있나

cmux는 2026년 초에 나온 macOS 전용 터미널입니다. AI 코딩 에이전트를 여러 개 동시에 돌리려는 사람들을 위해 만들어졌고, 화면 렌더링은 libghostty를 씁니다. wmux는 같은 발상을 Windows로 옮긴 물건입니다. 둘 다 최근에 등장했고, 사용자가 빠르게 늘고 있습니다.

### 왜 지금인가

세 가지가 겹쳤습니다.

**첫째, 테마를 바꾸려면 설정 파일을 직접 손대야 합니다.** cmux의 터미널 색은 Ghostty 설정 파일(`~/.config/ghostty/config`)에서 읽어옵니다. 색 하나를 바꾸려면 `palette = 4=#61afef` 같은 줄을 손으로 써야 하고, 16색 ANSI 팔레트를 전부 채우려면 스무 줄 가까이 됩니다. 결과가 어떻게 생겼는지는 저장하고 리로드해야 알 수 있습니다.

**둘째, 사람들이 실제로 여기서 막히고 있습니다.** cmux 저장소에는 "배경색, 글자색, 커서, 선택 영역의 RGB 값을 어떻게 지정하느냐"는 질문이 이슈로 올라와 있습니다. 또 다른 이슈는 테마 선택기가 앱 외형(밝게/어둡게)과 터미널 색상표를 구분하지 못해서, 사용자가 공들여 지정한 색을 조용히 덮어쓴다고 지적합니다.

**셋째, 두 터미널이 같은 언어를 씁니다.** cmux와 wmux 모두 Ghostty 테마 형식을 따릅니다. 색 키 이름(`background`, `foreground`, `cursor-color`, `palette`)이 같습니다. 편집기를 하나만 만들어도 두 터미널을 동시에 지원할 수 있다는 뜻입니다. 이것이 이 제품이 지금 가능해진 이유입니다.

## 4. 목표 (Objective)

### 무엇을, 왜

**설정 파일 문법을 배우지 않고도 자기 터미널 테마를 만들 수 있게 한다.**

터미널을 꾸미는 일은 취향의 영역입니다. 그런데 지금은 취향을 표현하기 전에 문법을 먼저 배워야 합니다. 이 순서를 뒤집는 것이 목표입니다. 색을 먼저 고르고, 문법은 결과물로 받아갑니다.

### 고객에게 주는 이익

- 색을 고르는 즉시 결과를 봅니다. 저장하고 리로드하는 왕복이 사라집니다.
- 좋아하는 사진이나 스크린샷에서 색을 뽑아 팔레트를 만들 수 있습니다.
- 자기 OS에 맞는 설명만 봅니다. macOS 사용자가 Windows 명령어를 읽을 일이 없습니다.

### 회사/제품 전략과의 연결

터미널 테마 편집기는 cmux·wmux 생태계로 들어오는 입구입니다. 테마를 만든 사람은 그 터미널을 계속 씁니다. 진입 장벽을 낮추는 일이 곧 사용자 유지로 이어집니다.

## 5. 시장 (Market Segments)

시장은 나이나 직업이 아니라 **사람들이 겪는 문제**로 나눕니다.

### 주 대상 — "내 터미널이 내 것 같지 않다"

cmux나 wmux를 이미 쓰고 있고, 기본 테마(wmux는 Dracula)가 마음에 들지 않는 사람. 번들된 450여 개 테마를 훑어봤지만 딱 맞는 게 없었습니다. 직접 만들고 싶은데 설정 파일을 열면 무엇부터 손대야 할지 모릅니다.

### 부 대상 — "색은 정했는데 옮기질 못하겠다"

이미 좋아하는 색 조합이 있습니다. 다른 에디터의 테마이거나, 좋아하는 이미지의 색이거나, 회사 브랜드 컬러입니다. 이걸 터미널로 옮기고 싶은데, 16색 ANSI 팔레트에 어떻게 배치해야 할지 감이 오지 않습니다.

### 제약 조건

- **cmux는 macOS 전용, wmux는 Windows 전용입니다.** Linux 사용자는 대상이 아닙니다.
- **cmux의 테마는 앱 전체에 적용됩니다.** 작업 공간(workspace)별로 다른 테마를 주는 기능은 아직 없습니다. 이 기능은 요청 이슈로만 올라와 있는 상태입니다. 따라서 편집기도 "테마 하나"를 만드는 도구로 설계합니다.
- 사용자는 편집기에서 만든 결과를 **직접 자기 컴퓨터에 붙여넣어야 합니다.** 웹에서 사용자 파일 시스템에 바로 쓸 수 없습니다.

## 6. 가치 제안 (Value Propositions)

### 해결하려는 일 (Jobs to be Done)

| 사용자가 하려는 일 | 지금의 고통 | Mux Editor가 주는 것 |
| --- | --- | --- |
| 터미널 색을 내 취향대로 바꾼다 | 설정 파일 문법을 배워야 함 | 색을 클릭하면 끝. 문법은 결과물로 나옴 |
| 바꾼 색이 어떻게 보이는지 확인한다 | 저장 → 리로드 → 확인의 왕복 | 가상 터미널에 즉시 반영 |
| 좋아하는 이미지의 색을 쓴다 | 스포이드로 하나씩 찍어 HEX를 옮겨 적음 | 이미지를 올리면 팔레트 자동 추출 |
| 만든 테마를 실제로 적용한다 | 파일 위치와 문법을 검색해야 함 | 내 OS에 맞는 파일 + 명령어를 그대로 복사 |

### 경쟁 대비 우위

기존 선택지는 세 가지입니다. 각각의 한계가 우리의 자리입니다.

- **번들 테마 목록 훑어보기** — 빠르지만 내 색이 아닙니다. 450개 중에 없으면 끝입니다.
- **설정 파일 직접 편집** — 완전한 자유가 있지만, 문법을 알아야 하고 결과를 미리 볼 수 없습니다.
- **범용 터미널 테마 생성기** — 미리보기는 되지만 cmux/wmux 형식으로 내보내주지 않습니다.

Mux Editor는 **자유도(직접 편집 수준)와 즉시성(미리보기)을 동시에** 제공하면서, cmux/wmux 형식으로 정확히 내보내는 유일한 도구를 목표로 합니다.

### 가치 곡선 (Value Curve)

경쟁자 대비 의도적으로 높이는 축과 낮추는 축입니다.

**높인다**: 미리보기 충실도 · 내보내기 정확성 · OS별 안내의 구체성 · 색 추출의 편의
**낮춘다**: 기능 개수 · 계정/로그인 · 테마 공유 마켓 (초기 버전에서는 만들지 않음)

### 7.2 주요 기능 (Key Features)

#### F1. OS 선택 (랜딩)

사용자가 macOS 또는 Windows를 고릅니다. 이 선택이 이후 모든 안내 문구, 파일 경로, 명령어를 결정합니다. 브라우저의 User-Agent로 기본값을 추천하되, 사용자가 언제든 뒤집을 수 있어야 합니다. 자동 감지를 강요하지 않습니다.

#### F2. 가상 터미널 미리보기

실제 터미널이 아니라, 터미널처럼 보이는 HTML입니다. 다음을 보여줍니다.

- 16색 ANSI 팔레트를 모두 쓰는 현실적인 출력 (`git status`, 테스트 결과, 로그 등)
- 배경, 글자, 커서, 선택 영역 색
- 최소 세 개의 시나리오 탭: 일반 셸 / 분할 패널 / AI 에이전트 세션 — cmux·wmux 사용자가 실제로 보는 화면이기 때문입니다

색상 변경은 리렌더 없이 CSS 변수 교체로 반영합니다.

#### F3. 테마 편집 패널

두 가지 입력 방식을 모두 지원합니다.

- **고른다**: 프리셋 목록에서 시작점을 잡고 손봅니다.
- **적는다**: HEX 값을 직접 타이핑합니다. 붙여넣기도 됩니다.

편집 가능한 항목은 cmux/wmux가 실제로 읽는 키로 제한합니다.

| 키 | 필수 | 설명 |
| --- | --- | --- |
| `background` | ✅ | 터미널 배경색 |
| `foreground` | ✅ | 기본 글자색 |
| `palette` 0–15 | ✅ | ANSI 16색 |
| `cursor-color` | — | 커서 색 |
| `cursor-text` | — | 커서 아래 글자색 |
| `selection-background` | — | 선택 영역 배경 |
| `selection-foreground` | — | 선택 영역 글자 |
| `background-opacity` | — | 배경 투명도 (0.0–1.0) |

여기 없는 키는 편집기에서 다루지 않습니다. 내보낸 파일이 실제로 동작하는 것이 최우선입니다.

#### F4. 색 추출

이미지를 올리면 대표 색을 뽑아냅니다. 세 가지 방식을 제공합니다.

- **자동 팔레트**: 이미지의 주요 색을 군집화해서 16색 슬롯에 배치합니다. ANSI 규약(0=검정, 1=빨강, 2=초록 … 9–15=밝은 계열)을 지키도록 색상환(hue) 기준으로 매칭합니다.
- **스포이드**: 이미지의 특정 지점을 클릭해서 색 하나를 뽑습니다.
- **슬롯 지정**: 뽑은 색을 배경/글자/특정 팔레트 번호에 직접 꽂습니다.

모든 처리는 브라우저 안에서 끝납니다. 이미지는 업로드되지 않습니다.

#### F5. 대비 경고 (접근성)

배경과 글자색의 명도 대비가 낮으면 경고합니다. 팔레트 색이 배경에 묻히는 경우도 표시합니다. 막지는 않습니다. 알려주기만 합니다. 예쁜 테마를 만들었는데 정작 글자가 안 보이는 일이 가장 흔한 실패입니다.

#### F6. 설정 추출

선택한 OS에 맞는 결과물을 만듭니다.

**macOS / cmux**
1. 테마 파일 내용 → `~/.config/ghostty/themes/<이름>` 에 저장
2. `theme = <이름>` 한 줄 → `~/.config/ghostty/config` 에 추가
3. `cmux reload-config` 실행 (또는 `⌘⇧,`)

밝은 테마와 어두운 테마를 함께 만든 경우, `theme = dark:<어두운이름>,light:<밝은이름>` 형식으로 안내합니다.

**Windows / wmux**
1. 테마를 `~/.wmux/config.toml` 의 색상 스킴으로 정의
2. 또는 Ghostty 형식 파일로 내보낸 뒤 `wmux config import-ghostty <경로>` 로 가져오기

각 블록에는 복사 버튼을 답니다. 테마 파일은 내려받기도 가능합니다.

### 7.3 기술 (Technology)

- **클라이언트 전용.** 서버도, 데이터베이스도, 로그인도 없습니다. 사용자의 테마는 브라우저 로컬 저장소에 남습니다. 이미지도 색상값도 서버로 나가지 않습니다.
- **미리보기는 진짜 터미널 에뮬레이터가 아닙니다.** 색을 보여주는 것이 목적이므로 HTML + CSS 변수로 충분합니다. 실제 PTY를 붙이지 않습니다.
- **테마 형식은 Ghostty를 따릅니다.** cmux와 wmux 모두 Ghostty 테마 문법(`key = value`, `palette = N=#RRGGBB`)을 읽습니다. 내부 데이터 모델을 이 형식 하나로 두고, OS별 차이는 "어디에 넣고 어떻게 리로드하는가"라는 안내 문구에서만 갈라집니다.
- **색 추출**은 Canvas API로 픽셀을 읽고 군집화합니다. 외부 라이브러리 의존을 최소화합니다.

### 7.4 가정 (Assumptions)

**검증되지 않은 믿음입니다. 착수 전에 확인해야 합니다.**

| # | 가정 | 왜 위험한가 | 확인 방법 |
| --- | --- | --- | --- |
| A1 | 우리가 지원할 wmux가 어느 프로젝트인지 확정되어 있다 | **`wmux`라는 이름의 프로젝트가 여러 개 존재합니다.** cmux를 Windows로 포팅했다는 저장소, tmux 대안을 표방하는 저장소, Rust 크레이트 등이 각각 따로 있습니다. 설정 파일 경로와 문법이 서로 다를 수 있습니다 | **최우선.** 대상 저장소 하나를 명시적으로 정하고, 실제로 설치해서 `~/.wmux/config.toml` 구조를 눈으로 확인 |
| A2 | 사용자가 설정 파일을 손으로 붙여넣는 것을 감수한다 | 웹앱은 사용자 파일을 직접 못 씁니다. 이 마찰이 KR1(40% 전환)을 깎을 수 있습니다 | 5명 사용성 테스트. 추출 페이지에서 이탈하는지 관찰 |
| A3 | 이미지에서 뽑은 색이 쓸 만한 ANSI 팔레트가 된다 | 사진의 주요 색 16개가 "빨강/초록/노랑" 같은 ANSI 역할과 맞아떨어진다는 보장이 없습니다. 결과가 엉망이면 이 기능은 장식품이 됩니다 | 프로토타입으로 이미지 20장을 돌려보고 결과를 눈으로 평가 |
| A4 | 미리보기 화면이 실제 터미널과 충분히 비슷하다 | 폰트, 자간, 안티앨리어싱 차이로 색이 다르게 느껴질 수 있습니다 | 같은 테마를 실제 cmux에 적용해 스크린샷 나란히 비교 |
| A5 | cmux의 테마 선택기가 우리 설정을 덮어쓰지 않는다 | cmux에는 앱 외형(밝게/어둡게) 선택이 사용자 지정 색을 조용히 덮어쓴다는 알려진 이슈가 있습니다 | 실제로 재현해보고, 재현되면 추출 페이지에 경고 문구 추가 |
| A6 | 테마 공유 기능 없이도 사용자가 만족한다 | 만든 테마를 남에게 보여주고 싶은 욕구가 클 수 있습니다 | 초기 사용자에게 직접 질문 |

## 8. 릴리스 (Release)

정확한 날짜 대신 상대적 기간으로 적습니다.

### 0단계 — 착수 전 (약 3일)

가정 A1을 해소합니다. 지원할 wmux 프로젝트를 확정하고, macOS와 Windows에서 각각 손으로 테마를 하나씩 적용해봅니다. **이 과정에서 알아낸 실제 파일 경로와 문법이 이 PRD의 7.2·7.4절을 덮어씁니다.** 여기서 밝혀지는 사실에 따라 범위가 달라질 수 있습니다.

### v0.1 — 첫 버전 (약 3~4주)

세 페이지가 다 있고, macOS(cmux) 하나만 제대로 동작합니다.

- 랜딩 페이지 + OS 선택 (F1)
- 가상 터미널 미리보기, 시나리오 탭 1개 (F2)
- 테마 편집 패널, 필수 키 전부 + 선택 키 (F3)
- cmux용 설정 추출 (F6)

Windows를 고른 사용자에게는 "곧 지원합니다"를 보여줍니다. 정직한 편이 낫습니다.

### v0.2 — Windows와 색 추출 (약 +2~3주)

- wmux 설정 추출 (F6)
- 이미지 색 추출 (F4)
- 대비 경고 (F5)

### v1.0 — 다듬기 (약 +2주)

- 미리보기 시나리오 3종 (분할 패널, 에이전트 세션)
- 밝은/어두운 테마 쌍 만들기 → `theme = dark:...,light:...` 내보내기
- 로컬 저장소에 테마 여러 개 보관
- 테마 파일 내려받기

### 이후 (범위 밖)

지금은 만들지 않지만, 사용자가 원하면 검토합니다.

- 기존 설정 파일을 불러와서 편집하기 (import)
- 만든 테마를 링크로 공유하기
- 다른 에디터 테마(VS Code 등)를 터미널 팔레트로 변환
- Linux 터미널 지원 — cmux·wmux 어느 쪽도 Linux를 지원하지 않으므로 별도 제품에 가깝습니다
- 작업 공간(workspace)별 테마 — cmux가 이 기능을 먼저 지원해야 가능합니다

## 확인이 필요한 사항 (Open Questions)

1. **어느 wmux인가?** 동명의 프로젝트가 여럿입니다. 하나를 정해야 7.2절의 Windows 내보내기 스펙을 확정할 수 있습니다. (가정 A1)
2. **테마 이름을 사용자가 짓게 할 것인가?** 파일명이 되므로 공백·대소문자 규칙이 필요합니다.
3. **미리보기에 쓸 폰트는?** 실제 터미널과 비슷해 보이려면 등폭 폰트 선정이 중요합니다.
4. **KR1의 40%는 어디서 나온 숫자인가?** 근거 없는 목표치입니다. 출시 후 실측으로 교체해야 합니다.
